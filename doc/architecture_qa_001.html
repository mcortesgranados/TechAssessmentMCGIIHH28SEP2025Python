Types of Services in Hexagonal/Clean Architecture

üîé Why do all three exist?

Each type serves a distinct and necessary function in medium/large systems or when you want high maintainability and scalability:

  • Domain Services domain/services/
    Encapsulate pure business logic.
    Do not depend on any technical details (no database, frameworks, or external APIs).
  • Application Services application/services/
    Orchestrate processes/use cases.
    Decide the flow, validate, coordinate entities, repositories, and events.
    Handle application business rules, but not pure domain rules.
  • Infrastructure Services adapters/services/ or adapters/events/
    Implement technical details: database access, email sending, integration with external APIs, event publishing, etc.
    Are the adapters for the interfaces (ports) defined by your domain/application.
üß© When do you use each?
  • If your logic is universal and model-specific → Domain
  • If you need to orchestrate several steps, validate, transform, or apply application business logic → Application
  • If the logic depends on technology or integration → Infrastructure

You keep each layer independent, easy to test and evolve.
You can change the infrastructure without altering the domain or application.
The domain stays clean and focused, the application flexible, and the infrastructure decoupled.
